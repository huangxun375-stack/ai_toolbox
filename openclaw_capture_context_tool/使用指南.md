# OpenClaw Context-Engine 可观测工具 — 使用指南

> 适用版本：基于 lossless-claw v0.4.0 的 LCM 诊断观测
> 最后更新：2026-03-21

---

## 一、概述
本工具用于观测 OpenClaw 的 context-engine 插件（当前为 lossless-claw）的运行行为，包括：

- **HTTP 流量抓包**：通过 mitmproxy 捕获 Gateway 与上游模型之间的所有请求/响应
- **LCM 诊断分析**：解析 lossless-claw 产生的诊断日志（bootstrap、ingest、assemble、compaction 全流程）
- **Web UI 可视化**：在浏览器中查看对话轨迹、LCM 内部事件时间线、token 用量对比
- **命令行诊断**：通过 `diag` 命令快速查看压缩行为

## 三、快速开始
### 3.1 一键安装
```bash
cd /path/to/openclaw_capture_context_tool

# 自动检测环境、创建 venv、安装依赖、生成 .env
./openclaw_capture_toolkit.sh setup
```

`setup` 会检查以下项目并报告状态：
- [OK] / [MISSING] Python 3.10+
- [OK] / [MISSING] mitmdump
- [OK] 创建 `.venv` 并安装依赖
- [OK] 复制 `env.example` → `.env`
- [OK] / [WARN] 端口 18080、8000 是否可用
- [OK] / [INFO] OpenClaw 二进制
### 3.2 编辑配置

```bash
vim .env
```

关键配置项：

```bash
# Gateway 地址（根据你的 OpenClaw 端口调整）
GATEWAY_BASE_URL=http://127.0.0.1:18789

# LCM 诊断文件路径（通常不需要改）
# LCM_DIAGNOSTICS_PATH=~/.openclaw/lcm-diagnostics.jsonl

# 代理端口（mitmproxy 监听）
MITM_PORT=18080

# Web UI 端口
CAPTURE_API_PORT=8000
```

### 3.3 启动

```bash
# 方式一：一键启动抓包栈 + 托管 Gateway
./openclaw_capture_toolkit.sh up

# 方式二：分步启动（如果 Gateway 已在运行）
./openclaw_capture_toolkit.sh start     # 启动 mitmproxy + API
# 然后手动设置 Gateway 的代理环境变量：
# HTTP_PROXY=http://127.0.0.1:18080
# HTTPS_PROXY=http://127.0.0.1:18080
# NODE_TLS_REJECT_UNAUTHORIZED=0
```

### 3.4 使用

1. **打开 Web UI**：浏览器访问 `http://127.0.0.1:8000/`
2. **进行对话**：在 Cursor 或其他客户端中与 Gateway 交互
3. **查看结果**：Web UI 自动刷新，展示对话轨迹和 LCM 诊断

### 3.5 停止

```bash
./openclaw_capture_toolkit.sh down      # 停止 Gateway + 抓包栈
# 或
./openclaw_capture_toolkit.sh stop      # 仅停止抓包栈
```

## 四、命令参考
### openclaw_capture_toolkit.sh

| 命令 | 说明 |
|------|------|
| `setup` | 检测环境、安装依赖、生成 .env |
| `start` | 启动 mitmproxy + Capture API |
| `stop` | 停止 mitmproxy + Capture API |
| `status` | 显示进程状态和数据统计 |
| `clear` | 清空抓包数据文件 |
| `up` | start + gateway-start |
| `down` | gateway-stop + stop |
| `request` | 发送一条测试请求到 Gateway |
| `gateway-start` | 启动托管 Gateway（带代理+cache-trace） |
| `gateway-stop` | 停止托管 Gateway |
| `gateway-status` | 查看托管 Gateway 状态 |
| `proxy-env` | 输出代理环境变量 |
| `proxy-run` | 在代理环境下运行命令 |
| `export-offline` | 导出为独立 HTML 文件 |
| `diag` | 解析并显示 LCM 诊断日志 |
| `instructions` | 打印快速使用步骤 |

### diag 命令详解

```bash
# 查看全部诊断
./openclaw_capture_toolkit.sh diag

# 按 sessionId 过滤
./openclaw_capture_toolkit.sh diag --session lcm-test

# 只看第 2 轮诊断
./openclaw_capture_toolkit.sh diag --round 2

# 只看压缩相关阶段
./openclaw_capture_toolkit.sh diag --stage compaction_evaluate,compact_result,leaf_summary

# 输出原始 JSON（可 pipe 给 jq）
./openclaw_capture_toolkit.sh diag --raw | jq .

# 从 stdin 读取（兼容旧方式）
cat ~/.openclaw/lcm-diagnostics.jsonl | python3 diag_cli.py --stdin --round 1

# 指定文件路径
python3 diag_cli.py /path/to/lcm-diagnostics.jsonl --session my-session
```

## 五、Web UI 功能说明

### 5.1 对话轨迹视图

打开 `http://127.0.0.1:8000/` 后可以看到：

- **左侧面板**：所有捕获的对话轨迹列表
- **右侧主区域**：选中轨迹的详细时间线

每个对话轮次包含：
- 用户输入 → Gateway → 模型请求 → 模型响应（含 SSE 流式）
- LCM 诊断事件（自动关联到对应轮次）
### 5.2 LCM 诊断面板

每个轮次下方展示该轮的 LCM 内部事件：
| 阶段 | 含义 |
|------|------|
| Bootstrap 开始/导入/结果 | 会话首次加载时的消息导入 |
| afterTurn 回调 | 每轮对话结束后的回调入口 |
| 消息持久化 (ingest) | 每条消息的 token 计数和存储 |
| 压缩评估 | 是否触发压缩（currentTokens vs threshold） |
| 叶子压缩详情 | 原始消息 → 摘要的压缩过程 |
| 叶子摘要 | 压缩前后 token 变化 |
| DAG 聚合摘要 | 多级摘要合并 |
| 上下文装配 | 最终发送给模型的上下文组成 |

### 5.3 Token 用量对比

`/api/compare/memory-tokens` 端点提供跨轮次的 token 用量对比。

## 六、LCM 诊断阶段速查表
| 阶段 | 产生位置 | 关键字段 |
|------|----------|----------|
| `bootstrap_entry` | engine.ts | `sessionFile`, `sessionKey` |

| `ingest` | engine.ts | `role`, `seq`, `tokenCount`, `contentPreview` |
| `afterTurn_entry` | engine.ts | `totalMessages`, `newMessageCount`, `messages` |
| `assemble_skip` | engine.ts | `reason` |
| `assemble_input` | engine.ts | `messagesCount`, `inputTokenEstimate`, `tokenBudget` |
| `assemble_output` | engine.ts | `outputMessagesCount`, `estimatedTokens`, `savingPct` |
| `context_assemble` | assembler.ts | `summaryCount`, `freshTailCount`, `tailTokens` |
| `compaction_evaluate` | compaction.ts | `currentTokens`, `threshold`, `shouldCompact` |
| `compact_skip` | compaction.ts | `reason`, `tokensBefore` |
| `compact_phase` | compaction.ts | `phase`, `status`, `reason` |
| `compact_result` | compaction.ts | `tokensBefore`, `tokensAfter`, `tokensSaved` |
| `leaf_pass_detail` | compaction.ts | `inputMessageCount`, `inputTokens`, `outputTokens` |
| `leaf_summary` | compaction.ts | `tokensBefore`, `tokensAfter`, `savingPct` |
| `dag_aggregate` | compaction.ts | `tokensBefore`, `tokensAfter`, `savingPct` |

## 七、API 端点参考
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/` | Web UI 首页 |
| GET | `/api/timeline` | 全部对话轨迹列表 |
| GET | `/api/trace/{trace_id}` | 单个轨迹详情 |
| GET | `/api/compare/memory-tokens` | Token 用量对比 |
| GET | `/api/lcm-diagnostics` | LCM 诊断数据 |

### `/api/lcm-diagnostics` 查询参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `session_id` | string | 按 sessionId 过滤 |
| `stage` | string | 按 stage 过滤（逗号分隔，如 `compaction_evaluate,compact_result`） |
| `after_ts` | int | 只返回 ts > after_ts 的记录（用于增量拉取） |

示例：
```
GET /api/lcm-diagnostics?session_id=lcm-test&stage=compaction_evaluate,compact_result
GET /api/lcm-diagnostics?after_ts=1774089847605
```

## 九、常见调试场景
### 场景 1：验证压缩是否触发
```bash
./openclaw_capture_toolkit.sh diag --stage compaction_evaluate
```

关注 `shouldCompact` 和 `currentTokens` vs `threshold` 的关系。
### 场景 2：查看某一轮的完整 LCM 流程

```bash
./openclaw_capture_toolkit.sh diag --round 2
```

可以看到该轮从 afterTurn 到 assemble 到 compaction 的完整流水。
### 场景 3：对比压缩前后 token 变化

```bash
./openclaw_capture_toolkit.sh diag --stage leaf_summary,dag_aggregate,compact_result
```

### 场景 4：导出分析结果分享
```bash
./openclaw_capture_toolkit.sh export-offline --output ./report.html
```

生成独立 HTML 文件，可直接发送给他人在浏览器中查看。

## 十、故障排除
| 问题 | 排查步骤 |
|------|----------|
| Web UI 无数据 | 1) `status` 检查进程 2) 确认 Gateway 使用了代理 3) `clear` 后重试 |
| LCM 诊断为空 | 1) 确认 `LCM_DIAGNOSTICS_ENABLED` 不是 `false` 2) 检查 `~/.openclaw/lcm-diagnostics.jsonl` 是否有内容 |
| LCM 事件关联到错误轮次 | 确认 lossless-claw 已更新（sessionId 统一修复） |
| `setup` 报 mitmdump MISSING | `pip install 'mitmproxy>=11.0.0'`（建议安装到系统或 venv 外） |
| 端口被占用 | 修改 `.env` 中的 `MITM_PORT` 或 `CAPTURE_API_PORT` |
